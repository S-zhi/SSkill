# Insight Report · 认知提升报告

把一段口播视频（当前主要是抖音）变成一份 1~2 页、能提升认知的速读报告。
核心链路：**抓取 ASR → 说话人拆分 → 四部分报告 → Markdown → PDF**。

## 目录结构

```
insight-report/
├── SKILL.md                      ← 你在这里（总流程）
├── scripts/
│   ├── fetch_douyin_asr.py       ← 跑 Coze 工作流，抓抖音 ASR 落盘
│   ├── build_pdf.py              ← Markdown → PDF（含中文字体、评级方块）
│   ├── verify_pdf.py             ← 交付前质检：结构（pypdf）+ 版面（pdfplumber），overlap/out_of_page 等
│   ├── render_pages.py           ← pdftoppm 把每页渲成 PNG，供人工视觉核查
│   ├── env_report.py             ← 记录/核对渲染器·依赖·字体版本
│   └── run_regression.py         ← 跑 tests/fixtures/ 下全部回归样例
├── references/
│   └── report-spec.md            ← 报告四部分的详细规格 + 完整示例 + PDF 渲染纪律（写报告/改 CSS 前必读）
├── assets/
│   ├── .env.example              ← Coze 令牌模板，用户需复制成 .env 填写
│   └── report.css                ← PDF 样式（流式容器、字号/行距下限，别手改）
├── tests/
│   └── fixtures/                 ← 排版回归样例：短文本/长文本/长链接/连续英文/两页分页
└── requirements.txt              ← 渲染链路依赖版本锁死，见文件内注释
```

## 何时用本 skill

用户给出抖音链接并想理解/提炼/做成报告，或直接说"生成认知报告 / 拆解这个视频 / 这段口播讲了啥"。若用户已自备 ASR 文本（不想跑抓取），跳过第 1 步，直接从第 2 步开始（对应下面的 `--asr-file` 参数）。

## 进度提示：贯穿全程的可观测性

每次**切换到一个新的步骤**时，先输出一行进度标识，再做那件事，格式固定为：

```
[第 X/6 步] 第 <N> 步 · <这一步具体在干什么，一句话>
```

- **X/6** 对应下面「端到端流程」的第 0～5 步，一一对应（第 0 步 → `1/6`……第 5 步 → `6/6`）。
- 示例：
  | 阶段 | 示例 |
  |---|---|
  | 第 0 步 | `[1/6] 第 0 步 · 解析参数并确认环境（.env / workflow-id / douyin-url）` |
  | 第 1 步 | `[2/6] 第 1 步 · 抓取 ASR（Coze 工作流运行中）` |
  | 第 2 步 | `[3/6] 第 2 步 · 说话人拆分` |
  | 第 3 步 | `[4/6] 第 3 步 · 写四部分报告` |
  | 第 4 步 | `[5/6] 第 4 步 · 转 PDF 并质检（结构/版面/视觉/环境四项）` |
  | 第 5 步 | `[6/6] 第 5 步 · 交付 report.pdf / report.md` |
- **粒度**：每进入一个新步骤报一次即可，不用每条命令都报；命令输出、报告正文照常展示，进度行只是加在前面的一行"路标"。第 4 步内部的三关检查（结构/版面自动检查 → 视觉渲染核查 → 环境记录）共用同一个 `[5/6]`，不必逐项再拆号；若因检查不过重跑，重跑那次仍复用 `[5/6]`。
- 这只是给已有的第 0～5 步加一行状态提示，**不改变**任何判断逻辑或执行顺序。

## 端到端流程

### 第 0 步：解析参数 & 准备环境（首次）

1. **解析命令行参数**：用户调用本 skill 时，除了给一个抖音链接，也可能带类似命令行的参数——支持以下几个，都是**可选**的，缺了就按规则回退或向用户索要，不瞎猜：
   - **`--douyin-url <抖音分享链接>`**：本次要处理的视频。
   - **`--workflow-id <Coze 工作流 ID>`**：不传 → 依次回退到 `.env` 里的 `COZE_WORKFLOW_ID`，再没有 → 回退到本文档记录的已知默认值 `7662042950008324137`。**这个默认值不是可以默认套用的"记忆"**——凡是本次没有用户显式传入、而是从 `.env` 或文档默认值带出来的工作流 ID，动手抓取前都要跟用户报一句"本次用工作流 ID `<id>`（来自 xxx），确认吗？"，**每次运行都要重新确认一遍，不能因为上次确认过就默认这次还适用**。
   - **`--asr-file <本地路径>`**：用户已有现成 ASR 文本时传入，跳过第 1 步，直接从第 2 步开始（对应上面"何时用本 skill"里的降级路径）。
   - **`--output <文件名>`**：ASR 落盘文件名，默认 `asr_raw.txt`，对应 `fetch_douyin_asr.py` 的同名参数。
   - 脚本本身还支持 `--api-token`（默认读 `.env` 的 `COZE_Personal_Access_Token`）、`--base-url`、`--input-text`（备用 JSON 入参）等，用户如果明确要用这些，透传给 `fetch_douyin_asr.py` 即可，不必都在 skill 层面重复定义——**这份参数列表允许用户自行追加**，新增一个参数时要说清楚它替代了流程里的哪个决策点、默认值是什么，并检查是否需要在下面某一步补一句"受此参数影响时如何处理"。
   - 参数都没传 → 完全按原有行为走：自然语言里找、找不到就向用户索要（工作流 ID / 抖音链接缺一不可，不能瞎猜）。

2. 安装依赖并确认 `.env` 已就绪：

   ```bash
   cd <skill-dir>
   pip install -r requirements.txt --break-system-packages -q
   ```

   Coze 抓取需要令牌。检查是否存在 `.env`：若没有，把 `assets/.env.example` 复制为 skill 根目录下的 `.env`，让用户填入 `COZE_Personal_Access_Token`（**不要替用户编造令牌**，缺失就明确告诉用户去填——这是硬性技术前提，不是可以商量跳过的审批，没令牌 Coze 调用就是跑不通）。

   ```bash
   [ -f .env ] || cp assets/.env.example .env
   ```

3. 确认运行所需的两样输入都已就绪：

   - **工作流 ID**（`--workflow-id`，或 `.env` 的 `COZE_WORKFLOW_ID`，或已确认过的文档默认值）
   - **抖音视频链接**（`--douyin-url`）——若走了 `--asr-file` 路径则不需要这个。

   若必需项缺一，先向用户索要，不要瞎猜。

### 第 1 步：抓取 ASR

**若第 0 步的参数解析拿到了 `--asr-file`，跳过本步**，直接读那个文件作为 ASR 文本，进入第 2 步。

否则执行：

```bash
python scripts/fetch_douyin_asr.py \
  --workflow-id "<用户的工作流ID>" \
  --douyin-url "<抖音分享链接>" \
  --output asr_raw.txt
```

脚本会累积工作流 MESSAGE 事件的 `content` 字段（ASR 文本主要落在这里），并写入 `asr_raw.txt`。

- `--workflow-id` 按第 0 步参数解析里的回退顺序取值（显式参数 > `.env` 的 `COZE_WORKFLOW_ID` > 文档默认值 `7662042950008324137`）；用的若不是用户本次显式传入的值，运行前要已经完成过第 0 步那句"确认吗"的核对，不要跳过这一步直接抓取。
- 若抓取为空，读脚本打印的事件日志排查（多半是工作流输出字段名、链接失效或工作流内部报错），把情况如实告诉用户，别硬编一份 ASR。

读回 `asr_raw.txt`（或 `--asr-file` 指向的文件）作为后续输入。

### 第 2 步：说话人拆分（理解用，不进正文）

通读 ASR，判断口播形态并拆分。**详细规则见 `references/report-spec.md` 的"说话人拆分"一节**，要点：

- **单播**（最常见）：一人到底 → 按句拆分、去口水词。
- **双播**：两人对谈 → 按 `A:/B:`（或身份）分轮。
- **多播**（极少）：三人及以上 → `A/B/C` 分轮。

这一步产物是"整洁的分句/分轮文本"，供你理解与后续摘录，不必全塞进报告。

### 第 3 步：写四部分报告（Markdown）

**动笔前先读 `references/report-spec.md`**，它给出每部分的精确规格和一个完整示例。四部分为：

1. **一、核心提炼** — 中心思想（引用块）+ 要点（≤3 点）+ 分区（约 5、不超 5、不过少）并给每分区打 `<span class="tag">` 标签。
2. **二、精华原文** — 从优化后的口播里截取有价值的故事/案例/整段，用 `<div class="excerpt">` 包裹，保留原味不改成白话。
3. **三、印证与批判** — 一句话总结 + 两句真实名人名言（引用块，避免纯白话）印证；先正向证明，发现纰漏则 `<div class="critique">` 批判；文末放 `<div class="rating">` 认同/不认同勾选方块。
4. **四、延伸应用** — 1~3 个方向，每方向 2~3 个具体 Case。

把报告写到 `report.md`。**成品控制在 1~2 页**：内容量参照上面的篇幅建议（摘录 1~3 段、Case 2~3 个/方向）来写；真超长就先砍第二部分摘录数和第四部分 Case 数，保住一、三部分。**这里的"1~2 页"是页数上限，不是必须挤进 1 页的下限**——写完转出来发现接近 1 页边缘、内容有点挤，就让它自然流到第 2 页，**不要**靠临时改小字号/行距/边距把它摁进 1 页，那正是 2026-07 那次引用框/ASR 框/批判框互相覆盖事故的根因。细节规则见 `references/report-spec.md` 的「PDF 渲染纪律」一节。

### 第 4 步：转 PDF 并质检（交付前的强制闸门，任一项不过禁止交付）

```bash
python scripts/build_pdf.py report.md report.pdf
```

生成后**必须依次跑完三关**，全过才能进入第 5 步交付；哪一关没过就回去改 `report.md`（内容太密就精简，别手改 CSS 应急），改完从这一步重跑：

1. **结构 + 版面自动检查**（pypdf 查页数/空页/ASR 摘录完整性；pdfplumber 查文字重叠、越界、字号、行距）：
   ```bash
   python3 scripts/verify_pdf.py report.pdf --md report.md --json report_verify.json
   ```
   验收线：`page_count` 在 1~2、`overlap=0`、`out_of_page=0`、`empty_pages=[]`、`asr_excerpts_missing=0`，且没有字号/行距相关的 fail 项。退出码非 0 = 没过，看 `issues` 字段里具体是哪几条。

2. **整页视觉渲染核查**（pdftoppm 出图，你自己用 Read 工具逐张肉眼看）：
   ```bash
   python3 scripts/render_pages.py report.pdf --outdir render
   ```
   逐张查：文字/区块有没有互相覆盖、有没有被裁切、有没有异常黑块、中文有没有变成方块/乱码、表格有没有被断成两半。这几类问题脚本不一定能可靠判定，必须肉眼确认。`pdftoppm` 缺失时脚本会打印按操作系统分类的安装指引。

3. **环境记录**（记一笔本次用的渲染器/依赖版本和字体探测结果，出问题时用于复现排查）：
   ```bash
   python3 scripts/env_report.py --append-log .report-env.log
   ```

三关全过 → 继续第 5 步。**任一关失败，禁止把当前的 `report.pdf` 交付给用户**——回去改 `report.md`（内容密度、摘录/Case 数量），修完从 `build_pdf.py` 重新跑一遍，不能跳过验证直接改小字号糊弄过去。

### 第 5 步：交付

用 `present_files` 把 `report.pdf` 和 `report.md` 一起交给用户（PDF 在前）。简短说明报告涵盖的中心思想与分区即可，不要长篇复述报告内容。

## 关键纪律

- **不编造**：令牌缺失、抓取失败、ASR 为空时如实说明，绝不虚构 ASR 或报告内容。
- **名言要真实**：第三部分引用的名言须真实准确、贴合思想，宁用经典不杜撰。
- **保原味**：第二部分是"摘录"不是"改写"，去口水词和轻微顺句可以，改成白话总结不行。
- **守篇幅，但不硬压**：1~2 页是页数上限，优先牺牲摘录与 Case 的数量来控制篇幅；**不通过缩小字号/压紧行距/收窄边距/负 margin/固定高度裁切来把内容摁进 1 页**——2026-07 的引用框/ASR 框/批判框互相覆盖事故就是这么来的。挤不下就让它流到第 2 页。
- **渲染引擎和字号/行距下限不临时改**：PDF 渲染器固定 WeasyPrint，依赖版本按 `requirements.txt` 锁死；正文字号 ≥9.5pt、行距 ≥1.45× 字号是硬指标，`assets/report.css` 已经配好，不要为了塞内容手动调小或手改 CSS 加内联样式，改动一律走 `references/report-spec.md` 的「PDF 渲染纪律」流程 + `scripts/run_regression.py` 回归测试。
- **交付前三关必须全过**：`verify_pdf.py`（结构+版面自动检查）→ `render_pages.py` + 肉眼核查（视觉渲染）→ `env_report.py`（环境记录），任一关失败都**禁止交付**，回去改 `report.md` 内容而不是绕过检查。
- **可扩展**：当前抓取环节是抖音专用（Coze 工作流）；未来接入其他来源时，只需替换第 1 步的抓取脚本，产出同样的纯文本 ASR，后续流程不变。
- **推断值要反复确认，不能"记住上次"**：`--workflow-id` 这类没有用户本次显式传入、而是靠 `.env` 或文档默认值带出来的参数，**每次运行都要重新向用户确认一遍**再用，哪怕上一轮确认过——不能因为"之前问过"就默认这次还是同一个答案。
- **进度提示不能省**：按第 0～5 步输出 `[X/6]` 进度标识，让用户随时知道跑到哪一步、要不要重跑第 4 步精简篇幅。
