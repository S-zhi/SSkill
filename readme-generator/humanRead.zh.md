# README Pipeline 设计思路

本文档用于在用户询问“为什么 README 要这样生成”“当前 Pipeline 的设计思路是什么”时，帮助 AI 用简洁中文回答。

---

# 1. 核心思想

本 Skill 的 README 生成逻辑不是传统模板填空，而是按照读者进入项目的路径组织。

路径是：

```text
先知道项目是什么
↓
再快速跑起来
↓
然后学会配置和使用
↓
再理解项目结构
↓
最后参与开发、贡献、升级和维护
```

所以 README 不应该只是“项目简介 + 安装命令 + 许可证”的平铺结构，而应该是一个项目入口。

---

# 2. README 的定位

README 是项目的门面和入口，但不是所有文档的集合。

它应该承担：

- 介绍项目
- 引导快速开始
- 指向配置和命令说明
- 指向架构和开发文档
- 指向贡献、安全、版本和许可证信息

它不应该承担：

- 完整 PRD
- 完整技术设计文档
- 完整 API 文档
- 完整测试文档
- 完整 AI Prompt 或 Skill 规则

这些内容应拆到独立文档中，README 只做索引。

---

# 3. 为什么先写产品介绍

用户进入项目时，第一件事不是看安装命令，而是判断：

- 这个项目是什么？
- 能解决什么问题？
- 是否值得继续看？

所以 README 开头需要有项目定位、功能展示、平台与技术支持。

其中平台、依赖版本、技术栈建议使用 Shields.io Badge，因为它比纯文字更直观。

---

# 4. 为什么快速开始必须有验证步骤

很多 README 只写启动命令，但没有告诉用户怎样判断项目是否真的启动成功。

这会导致用户运行完命令后仍然不确定：

- 服务有没有起来？
- 页面地址是什么？
- 接口是否正常？
- CLI 是否安装成功？

所以快速开始必须包含“验证启动成功”。

例如：

- Web 项目访问页面
- 后端服务请求 `/health`
- CLI 项目运行 `--version` 或 `help`
- Docker 项目运行 `docker compose ps`

---

# 5. 为什么配置与命令要单独成章

快速开始只解决“最短路径跑起来”。

但用户跑起来以后，还会想知道：

- 可以配置哪些环境变量？
- 配置文件怎么写？
- 支持哪些命令？
- 常用参数是什么？

所以第三章专门放“配置与命令说明”。

其中“全部命令”和“常用参数”应该合并，因为参数本来依附于命令，分开写会让读者来回跳。

---

# 6. 为什么系统架构和项目结构不要写太细

README 中的架构说明是为了帮助开发者快速建立整体印象，不是替代完整技术文档。

所以 README 中只需要：

- 系统架构图
- 简要说明
- 项目目录结构

不需要展开：

- 每个核心模块
- 详细数据流
- 每个关键文件
- 完整实现细节

这些应该放到 `docs/architecture.md` 或技术文档中。

---

# 7. 为什么要有开发者指南

仅仅能运行项目，不代表能安全提交代码。

开发者在提交 PR 前需要知道：

- 本地如何开发
- 提交前要跑哪些检查
- 云端 CI 会检查什么
- 哪些检查必须通过才能合并

这可以降低协作风险，也能减少 AI 自动提交代码导致的错误。

---

# 8. 为什么安全漏洞不能走公开 Issue

安全漏洞如果公开提交 Issue，可能会让攻击者在修复前看到漏洞细节。

所以 README 应该明确：

- 普通 Bug 可以走 Issue
- 安全漏洞不要走公开 Issue
- 安全漏洞应走安全披露邮箱或 `SECURITY.md`

---

# 9. 为什么要有版本管理

项目一旦有人使用，就会遇到升级问题。

版本管理告诉用户：

- 历史版本在哪里看
- 每个版本改了什么
- 是否存在破坏性更新
- 如何从旧版本升级到新版本

因此 README 应该提供：

- Release 与 Tag
- CHANGELOG
- Upgrade Guide

---

# 10. 为什么要支持双语 README

中文主 README 方便中文维护者修改和检查。

英文 README 方便更广泛的开发者、用户和开源社区阅读。

为了降低维护成本，应采用：

```text
README.md       # 中文主版本
README.en.md    # 英文同步版本
```

修改时先改中文，再只同步英文对应章节，不重翻整篇。

---

# 11. 为什么 AI 文档不直接写进 README

README 可以被 AI 阅读，但不应该承载大量 AI 专用指令。

原因是：

- README 面向所有读者。
- AI 指令面向具体工具。
- 混在一起会让 README 臃肿。
- 不同 AI 工具有不同文件约定。

所以 README 只链接：

- `AGENTS.md`
- `CLAUDE.md`
- `.github/copilot-instructions.md`
- `docs/ai.md`

具体 AI 规则放在这些文件中。

---

# 12. 一句话总结

这套 Pipeline 的核心是：

> README 不是一个简单说明书，而是整个项目文档体系的入口。它负责让人快速理解、运行、使用和参与项目，而详细内容应拆到独立文档中维护。
