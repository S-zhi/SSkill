# 已有 README 的检测与增量修改规则

本文档定义当项目中已经存在 README 时，应该如何修改，而不是直接覆盖。

---

# 1. 基本原则

如果仓库已有 README：

1. 不要直接覆盖。
2. 先分析已有内容。
3. 保留仍然正确、有价值的内容。
4. 删除或改写过时、重复、冲突的信息。
5. 按 Pipeline 补齐缺失章节。
6. 只修改需要修改的章节。
7. 修改中文 README 后，同步英文 README 的对应章节。

---

# 2. 检查 README 文件

优先检查：

```text
README.md
README.zh-CN.md
README.en.md
README_EN.md
readme.md
```

如果存在多个 README：

- 优先把 `README.md` 作为中文主 README。
- 如果 `README.md` 已经是英文，需要根据用户要求决定是否改为中文主 README。
- 英文内容迁移或同步到 `README.en.md`。

---

# 3. 分析已有 README

分析以下内容：

- 当前语言
- 项目定位是否清晰
- 是否有截图或 Demo
- 是否有 Badge
- 是否写明支持平台
- 是否写明依赖版本
- 是否有快速开始
- 是否有环境变量说明
- 是否有配置文件说明
- 是否有命令说明
- 是否有架构或目录结构
- 是否有开发者指南
- 是否有贡献指南
- 是否有安全漏洞报告方式
- 是否有版本管理和更新日志
- 是否有许可证
- 是否有维护者和帮助入口

---

# 4. 修改方式

## 4.1 保留

应保留：

- 真实项目介绍
- 已验证可用的安装命令
- 已存在的截图、GIF、Demo
- 已存在且有效的 Badge
- 正确的环境变量说明
- 正确的贡献流程
- 正确的 License 链接
- 正确的安全披露方式

## 4.2 补充

应补充：

- 缺失的平台与依赖版本
- 缺失的文档导航
- 缺失的验证启动成功步骤
- 缺失的基础 CLI 命令
- 缺失的配置说明
- 缺失的开发者检查命令
- 缺失的 CI 验证说明
- 缺失的版本管理说明
- 缺失的安全漏洞报告方式

## 4.3 合并

如果已有 README 中多个小节表达相近，应合并到 Pipeline 对应章节。

例如：

- “安装” + “运行” + “启动服务” → 合并到 “2. 快速开始”
- “参数说明” + “命令列表” → 合并到 “3.3 命令与参数说明”
- “目录结构” + “项目结构” → 合并到 “4.2 项目目录结构”

## 4.4 删除或改写

应删除或改写：

- 已失效链接
- 过时命令
- 不再支持的平台
- 错误依赖版本
- 公开暴露的 Token 或密钥
- 与项目实际不符的功能描述
- 重复出现的长段说明

---

# 5. 增量修改策略

只修改必要章节。

不要因为新增一小节而重写全文。

示例：

```text
用户要求增加安全漏洞报告方式
↓
只修改：
- 文档导航中的“安全漏洞报告”入口
- 6.4 安全漏洞报告
- README.en.md 对应章节
```

---

# 6. 双语同步

如果仓库已有双语 README：

1. 先修改中文主 README。
2. 再同步英文对应章节。
3. 不重翻整篇英文 README。

如果只有中文 README：

1. 保留并升级中文 README。
2. 新增 `README.en.md`。

如果只有英文 README：

1. 根据英文内容生成中文 `README.md`。
2. 将英文版本保留或迁移为 `README.en.md`。
3. 两个文件顶部都加入语言切换入口。

---

# 7. 冲突处理

如果 README 中的信息与项目文件冲突：

- 以项目文件为准。
- 例如 `package.json` 中是 `pnpm dev`，README 写 `npm start`，应优先相信 `package.json`。
- 例如 `.env.example` 有 8 个变量，README 只写 2 个，应补齐。
- 例如 `LICENSE` 是 Apache 2.0，README 写 MIT，应改为 Apache 2.0。

如果无法判断，标记为“需要补充”。

---

# 8. 修改说明

如果用户要求说明修改内容，应按以下格式简要回答：

```text
本次主要修改：

1. 补充平台与依赖版本 Badge。
2. 增加文档导航入口。
3. 补充快速开始中的验证步骤。
4. 增加开发者提交前检查。
5. 同步更新 README.en.md。
```

如果用户只要求输出文件，不要输出冗长解释。
