# Issue 格式规范（Bug Fix / Feature）

Phase 4 严格套用本文件的模板。标题格式固定，正文分段固定，缺段不合格。

---

## 一、Bug Fix

### 标题
```
【Bug Request】<一句话写清具体是什么问题>
```
- 一句话说清“哪里 + 什么问题”，让人扫标题就知道是什么 bug。
- 反例：`【Bug Request】有个 bug`（太空）。
- 正例：`【Bug Request】并发导出时 exporter 复用同一 buffer 导致输出内容串行错乱`。

### 正文（三部分，用小标题分隔）
```markdown
## 一、问题描述
<Bug 的具体问题：现象是什么、在什么条件下触发、期望行为 vs 实际行为。若能复现，给出复现步骤。>

## 二、相关配置
<触发这个 bug 相关的具体配置：运行环境 / 语言与依赖版本 / 相关配置项或参数 / 涉及的文件·模块·函数（挂代码证据）。>

## 三、修复建议
<如何解决这个问题的建议：根因分析 + 建议改法（可给伪代码/思路，不必写完整补丁）+ 可选的验证方式（注意验证必须幂等、不破坏数据）。>
```

### 完整示例
```markdown
【Bug Request】并发导出时 exporter 复用同一 buffer 导致输出内容串行错乱

## 一、问题描述
现象：同时发起多个导出请求时，部分导出文件里混入了其它请求的内容片段。
触发条件：并发 ≥ 2 个导出任务，且任务体积较大（触发多次写入）。
期望：每个请求的输出相互隔离，内容完整；实际：内容交叉污染。
复现：并发调用 `export()` 两次，对比输出与输入不一致。

## 二、相关配置
- 环境：Linux / Python 3.11
- 依赖：无特殊依赖
- 代码证据：`internal/export/exporter.py` 中 `Exporter` 为模块级单例，`self._buf` 在多个请求间被共享写入（`writer.py:L42` 处未加隔离）。

## 三、修复建议
- 根因：可变状态 `_buf` 挂在共享实例上，并发下相互覆盖。
- 建议：把 buffer 改为每次调用内的局部变量 / 每请求独立实例；或对写入路径加锁但优先无共享方案。
- 验证：写一个并发跑 N 次导出的单测，断言每份输出与其输入一致（用临时目录，幂等、不触外部状态）。
```

## 二、Feature

### 标题
```
【Feature】<一句话写清具体是什么需求>
```
- 一句话说清“要加什么能力”。
- 正例：`【Feature】导出支持流式写入，避免大文件一次性占满内存`。

### 正文（两部分，用小标题分隔）
```markdown
## 一、为什么要实现
<简单说明动机：对应哪个用户 / 什么场景 / 解决什么痛点，为什么现在值得做。1–3 句即可，讲清“为什么”。>

## 二、用户故事与收益
<从用户故事角度分析：作为<角色>，我想要<能力>，以便<目的>。
说清它真正解决了什么问题、能带来什么收益（体验/效率/成本/差异化其一或多个）。
可顺带一句实现可能性：现有架构是否支撑、大致改动量。>
```

### 完整示例
```markdown
【Feature】导出支持流式写入，避免大文件一次性占满内存

## 一、为什么要实现
当前导出把全部内容拼在内存里再落盘，处理大文件（数百 MB）时内存峰值高、易 OOM。
面向“批量导出大数据集”的用户，这是硬伤，值得优先做。

## 二、用户故事与收益
作为一个需要导出大数据集的用户，我想要导出过程边生成边写盘，以便不受单机内存限制、也不用担心大文件把服务拖垮。
收益：内存占用从“与文件大小成正比”降到常数级；支持更大文件；服务更稳。
实现可能性：现有 `Exporter` 已是分块写入结构，改成 `yield`/流式 writer 改动量中等，无需动数据模型。
```

## 格式自检
- 标题前缀严格是 `【Bug Request】` 或 `【Feature】`，一句话说清。
- Bug 三段齐全：问题描述 / 相关配置 / 修复建议。
- Feature 两段齐全：为什么要实现 / 用户故事与收益。
- 正文里“涉及的代码位置”尽量写具体（文件·模块·函数），这是证据也是可执行性的体现。
