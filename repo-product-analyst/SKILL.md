# Repo Product Analyst（代码仓库 → 产品价值分析）

把一个**已存在的代码仓库**翻译成一份产品分析报告：它解决了什么、做到了什么程度、接下来值得做什么。

核心理念：**先建图谱、再做分析**。直接读全部源码既慢又烧 Token；先用 GitNexus 把代码库索引成知识图谱，之后用图谱查询（依赖、调用链、影响面）精准定位关键代码，只读真正需要的部分。

**两条贯穿始终的原则：**
- **证据优先，拒绝空想**：每一个关于"项目做了什么 / 没做什么"的结论，尽量挂上具体代码证据（文件 / 模块 / 函数 / 入口）。说不清来源的结论宁可不写。
- **语言跟随用户**：报告默认用用户当前对话所使用的语言输出（中文用户输出中文）。

## 总体流程

按顺序执行 Step 1 → Step 6，最后按模板产出报告。

### Step 1 — 定位并确认目标仓库

1. 确认要分析哪个仓库。如果用户没指明、或当前目录下有多个候选，**先问清楚**再继续。
2. `cd` 到该仓库的**根目录**（GitNexus 的索引和后续命令都要从仓库根目录跑）。

### Step 2 — 用 GitNexus 建立知识图谱

1. 在仓库根目录执行索引：
   ```bash
   npx gitnexus analyze
   ```
   它会走一条多阶段流水线（Structure → Tree-sitter AST 解析 → 跨文件 Resolution → 聚类 → 执行流），把整个代码库索引成本地知识图谱。中型项目通常一分钟内完成。
   - 若 npm 11.x 下 `npx` 安装崩溃，改用：
     `pnpm --allow-build=@ladybugdb/core --allow-build=gitnexus --allow-build=tree-sitter dlx gitnexus@latest analyze`
   - 代码有较大改动后需要刷新索引：再次 `npx gitnexus analyze`（或 `gitnexus analyze --force` 全量重建）。
2. 确认索引成功：
   ```bash
   gitnexus status   # 当前仓库的索引状态
   gitnexus list     # 已索引的全部仓库
   ```
3. 检查 GitNexus MCP 是否可用：
   - **可用** → Step 3 优先使用 MCP 工具（query / context / impact / detect_changes / cypher）探索。
   - **未配置但需要** → 可一次性配置：`npx gitnexus setup`（自动写入编辑器的 MCP 配置）。⚠️ 这会修改全局/编辑器配置，属于会留下持久改动的操作，**执行前先向用户确认**。
   - **无法使用 MCP** → 不阻塞，直接走 Step 3 的**降级方案**。

> **降级方案（GitNexus 跑不起来时，如缺 node、安装失败、离线环境）：**
> 不要卡死。改为手动探索：读 `README` / `docs`、列目录树、找包管理清单（package.json / pyproject.toml / go.mod 等）、定位入口文件与核心模块、抽样阅读关键代码。后续 Step 同样适用，只是证据来自人工阅读而非图谱。

### Step 3 — 探索，建立理解

目标：搞清楚**这个项目是做什么的、给谁用的、架构长什么样、主流程怎么走**。

优先用 GitNexus MCP 工具（成本远低于逐文件读取）：
- `list_repos` → 选定目标仓库；
- `query "<概念/功能>"` → 混合检索，快速定位相关代码簇；
- `context <符号>` → 某个函数/类的 360° 视图（定义、调用方、被调用方）；
- `impact <符号>` → 某处改动的影响面/调用链（用来判断核心模块）；
- `cypher "<图查询>"` → 自定义图查询，挖掘架构关系。

同时阅读：README/文档、包管理清单与依赖、入口文件、关键配置。把"这个项目是什么"先在脑子里立起来，再进入产品分析。

### Step 4 — 写项目概述

两段式：
1. **一句话定位**（1–2 句，点出领域 + 核心能力 + 目标用户）。
   - 示例风格："这是一个专注于爬取主流社交媒体（抖音、B 站、知乎）的评论与子评论、UP 主元信息、视频描述信息的爬虫系统。"——但要比这更具体。
2. **详细概述**（一段）：领域 / 目标用户 / 核心能力 / 技术栈 / 架构形态。

### Step 5 — 分析【已交付】的产品价值：痛点 / 痒点 / 爽点

用户需求分三个层次，用它来判断这个项目**已经交付了哪一层的价值**：

- **痛点（Must-have）**：用户不得不解决的问题，不解决就持续痛苦、甚至愿意付费。是产品的立身之本。
- **痒点（Nice-to-have）**：让体验更好的"理想态"，有了很开心、没有也能忍。属于锦上添花。
- **爽点（Wow-factor）**：超出预期的体验，用户"哇"出来的那一刻，是差异化竞争力、是被主动推荐的理由。

对照代码现状，逐层列出**项目已经做到的**，每条尽量挂代码证据：
- **已解决的痛点** —— 它解决的核心问题是什么？证据：<对应模块/入口/函数>
- **已实现的痒点** —— 哪些是提升体验但非刚需的能力？证据：<...>
- **已实现的爽点** —— 有没有让人眼前一亮的差异化能力？证据：<...>

（注意：某些项目可能只到痛点层、还没有痒点/爽点，如实写"暂无"即可，不要硬凑。）

### Step 6 — 分析【未交付】的缺口与未来方向

基于同一套价值模型，找出项目**还没做、但值得做**的部分，并排优先级：
- **待解决的痛点（优先级最高）**：还有哪些核心问题没被覆盖？这是 MVP 之后最该补的。
- **可加入的痒点**：哪些体验优化能在迭代中逐步加上？
- **可打造的爽点（差异化）**：哪些"超预期"能力能成为这个项目被记住的理由？

每条都要落到**代码现状**：当前缺什么 / 现有架构是否支撑 / 大致改动量，给出"先做什么、后做什么"的建议排序（可参考 影响 × 成本 简单排序）。

## 输出报告模板

**严格按此结构输出**（标题可随项目调整，但层次保持一致）：

```markdown
# 项目产品分析：<项目名>

## 一句话概述
<1–2 句定位>

## 项目概述
<领域 / 目标用户 / 核心能力 / 技术栈 / 架构形态>

## 已交付的产品价值
### 痛点（Must-have，已解决）
- <能力描述> —— 证据：<文件/模块/函数>
### 痒点（Nice-to-have，已实现）
- <…>（若无则写"暂无"）
### 爽点（Wow，已实现）
- <…>（若无则写"暂无"）

## 未来可做的方向
### 待解决的痛点（优先级最高）
- <机会> —— 现状缺口：<…> —— 价值：<…>
### 可加入的痒点
- <…>
### 可打造的爽点（差异化）
- <…>

## 优先级建议
<把上面的"未来方向"排个序，每条一句话说明理由（影响 × 成本）>
```

报告默认直接在对话中给出。若用户需要落盘，可另存为仓库根目录下的 `PROJECT_ANALYSIS.md`（写文件前向用户确认一次）。

## 注意事项 / 边界

- **Token 纪律**：能用图谱查询解决的，不要 `cat` 整个目录。GitNexus 的价值就在于"精准定位、按需阅读"。
- **会留改动的操作要确认**：`gitnexus setup`（改全局/编辑器配置）、写 `PROJECT_ANALYSIS.md`（在用户仓库里建文件）——执行前都先征得用户同意。
- **图谱会过期**：仓库有大改动后，分析前先 `npx gitnexus analyze` 刷新一次，否则影响面/调用链会与 HEAD 不一致。
- **诚实优先**：分析的是真实代码，不是产品宣传稿。没做到的层级（如还没有爽点）就如实说，缺口部分基于代码事实推断，不夸大。
