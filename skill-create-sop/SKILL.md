# Skill 流程优化 SOP（参数化输入 + 全链路可观测性）

这不是一个"写业务逻辑"的 Skill，是一套**改造现有 Skill 流程结构**的方法论：把一个已经能跑的 SKILL.md，升级成"可带参数调用、执行过程可观测"的版本，同时不破坏它原有的判断逻辑和安全边界。

来源：在对 `github-issue-solver` 和 `repo-issue-hunter` 两个 Skill 做流程改造时反复用到、抽出来的通用套路。

**两条 SOP，缺一不可：**
1. **参数化输入**：流程最前面加一个"解析命令行参数"的环节，参数全部可选、可扩展，用户可自行追加；一旦涉及"跳过确认"的参数，边界必须和用户对齐，不能自己拍脑袋。
2. **全链路可观测性**：以`[第几步/共N步]`的形式，在切换到每个顶层阶段/子步骤时汇报进度。

---

## 什么时候用这个 SOP

- 用户明确要求"给这个 Skill 加参数""让这个 Skill 支持 xxx 参数"。
- 用户明确要求"这个 Skill 跑起来不知道到哪一步了，加个进度提示"。
- 用户说"参考上一个 Skill 的模式，把这个也改一下"——通常就是指这两条 SOP。
- 用户在设计一个新 Skill 的流程骨架时，希望从一开始就具备这两个能力。

**不要用于**：纯粹修改 Skill 的业务判断逻辑、格式模板、领域知识（比如改 issue 正文模板、改 bug 判定标准）——那些是各 Skill 自己的内容，不归这个 SOP 管。这个 SOP 只管"流程的外壳"：怎么传参、怎么报进度。

---

## SOP ① 参数化输入

### 1. 参数解析要放在流程最前面，独立成一个步骤

不要把参数解析散落在各个阶段里临时判断。在目标 Skill 现有的 Step 0 / Phase 0（或等价的"启动引导"阶段）里，新增一个明确的子步骤，比如"解析命令行参数"，统一在这里列出所有支持的参数、说明每个参数对应替代了流程里哪个原本靠自然语言判断或询问用户的决策点。

### 2. 参数必须可选，且向后兼容

- 每个参数都要能单独省略；全部不传时，流程要能退回原来的行为（自然语言里找 / 默认值 / 正常逐项确认），不能因为加了参数支持就让旧的调用方式失效。
- 参数之间尽量互相独立、可任意组合，避免强耦合（例如不要设计成"必须先传 A 才能传 B"）。

### 3. 参数命名要能一眼对应到流程里的判断点

命名用 `--xxx` 风格，语义上要能让人一看就知道它替代的是流程里的哪一步。常见几类：
- **定位类**（如 `--repo`）：替代"这次操作的目标是什么"这一步原本要靠自然语言识别或反问用户的环节。
- **筛选/范围类**（如 `--issue`、`--direction`）：替代"这次要处理哪些对象/关注哪个方向"，通常伴随一条规则——用参数指定的范围仍要过原有的合法性检查（不能因为是参数传入就绕过校验），检查不通过时要**弹警告、跳过、继续处理其余的**，而不是让整个流程中断。
- **模式切换类**（如 `--direction` 触发"定向核实"代替"全仓扫描"）：给出后流程应切换到更聚焦、成本更低的执行路径，同时要明确说清"切换后不再做的事情"（比如不再做开放式扫描），避免模式之间行为混淆。
- **跳过确认类**（如 `--skip`）：见下一条，这是最容易出问题的一类，单独展开。

### 4. "跳过确认"类参数：边界必须先和用户对齐，不能自己猜

这是参数化 SOP 里最容易踩坑的地方。改造一个 Skill 之前，先通读全文，找出所有"先向用户确认""需要用户同意""不能自动执行"之类的语句——这些就是候选的"确认点"。然后把它们分成两类：

- **真正的审批/确认**：动作本身没有对错之分，只是"要不要做"取决于用户意愿或风险偏好（比如推送分支、创建 issue、写入全局配置）。这类**可以**被"跳过确认"参数覆盖。
- **硬性技术前提**：不满足就是物理上做不下去，不是"用户要不要批准"的问题（比如登录鉴权、依赖工具是否安装）。这类**永远不能**被跳过——跳过了流程也执行不了，跳过参数对它无意义。

**这两类不总是显而易见，边界要靠具体分析，不能套模板。** 典型例子：github-issue-solver 里"gh 登录"是硬性前提（不受 `--skip` 影响），但 repo-issue-hunter 里"建 issue 前的用户确认"看起来风险更高（创建公开可见内容），却被明确要求和 `--skip` 完全对齐、一并跳过——这个判断改造者自己做不了主，是**问了用户才定下来的**。

因此：**遇到"跳过确认"类参数、且候选确认点的风险等级不是显然一致时，用 `AskUserQuestion` 向用户确认边界，不要自行假设。** 问题里要把候选的确认点逐条列出来，说明每条的风险性质，给出一个"保守（部分跳过）"选项和一个"和参照对象完全对齐（全部跳过）"选项，让用户选。

### 5. 用户可以自行追加参数——新增时要过一遍全文，不能只改定义处

用户提出要加一个新参数（比如"再支持一个 `--xxx`"）时，除了在参数解析步骤里写清楚它的语义、默认值、和已有参数的交互关系之外，**必须回头检查全文**：
- 这个新参数是否影响某个已有的确认点？影响的话，要在**那个确认点原本的位置**补一句"受此参数影响时如何处理"，不能只在参数定义处写一遍就当完事——用户执行到那一步时是照着那一节读的，不会回头翻参数总览。
- 是否需要更新 frontmatter 的 `description`？新参数如果是这个 Skill 的核心能力之一，要体现在 description 里，否则会被漏掉、也影响触发匹配。
- 是否需要更新文末的自检清单 / 参考文件列表？

### 6. 参数解析结果要"记下来，别重复问"

流程后续步骤用到某个参数解析出的值时，直接引用即可（如"Step 0.1 解析到 `--repo`"），不要在后面的步骤里又用自然语言重新问用户一遍同样的信息。

## SOP ② 全链路可观测性

### 1. 数出目标 Skill 的顶层阶段总数 N

顶层阶段通常已经是 Skill 文档现成的结构——`Step 0/1/2/3` 或 `Phase 0～6` 之类。**不要另起一套新编号**，直接复用文档已有的阶段划分作为 N，改造成本最低、也最不容易和原文档脱节。

### 2. 统一进度格式，独立成一个小节集中定义

不要把格式说明分散写在每个步骤里，在流程正文之前新增一个独立小节（例如"进度提示：贯穿全程的可观测性"），一次性定义：

```
[第 X/N 步] <阶段名> · <这一小步具体在干什么，一句话>
```

然后给一张"阶段/子步骤 → 示例进度行"的映射表，覆盖文档里所有会被切换到的子步骤，方便后续新增步骤时对照着补。

### 3. 粒度：按文档已有的子步骤编号级别汇报，不要更细也不要更粗

- **不要每条 bash 命令都报**——会刷屏，正常的命令输出、解释文字照常展示即可，进度行只是加在前面的一行"路标"。
- **不要几个大阶段合并成一行**——用户会看不出实际进展。
- 合适的粒度通常就是文档里已经存在的编号小节这一级（`2a/2b/2c`、`0.1/0.2/0.3` 这种）。如果目标 Skill 还没有这一级编号，先补上编号再挂进度提示。

### 4. 循环结构要体现"第几个/共几个"

如果流程里有对一批对象循环处理的环节（比如"对选中的每个 issue 循环执行"），进度行要嵌套体现循环位置，例如：

```
[3/4] Step 2 · issue 2/3（#15）· 2c 创建 linked branch
```

不能只报"处于 Step 2"，让用户看不出批量任务跑到第几个了。

### 5. 明确这只是"加一行提示"，不是流程重构

进度提示是纯粹的可观测性增强，**不改变**任何判断逻辑、分支条件或执行顺序。改造时要克制，不要借机顺手调整原有的业务逻辑——那是另一件事，应该拆开单独做、单独跟用户确认。

## 改造一个目标 Skill 的操作步骤

1. **定位目标 SKILL.md**：用户指定的、或者当前对话里正在讨论的那个 Skill。
2. **通读现有流程**：识别顶层阶段划分（数出 N）、找出全部"确认/审批"类语句、看现有是否已经有编号到子步骤这一级。
3. **应用参数化 SOP**：
   - 列出要新增的参数草案，标明每个参数对应流程里的哪个决策点。
   - 如果其中有"跳过确认"类参数，且候选确认点风险等级不完全一致 → 用 `AskUserQuestion` 和用户对齐边界，不要自行假设。
   - 在流程最前面新增"解析命令行参数"子步骤。
4. **应用可观测性 SOP**：新增独立的"进度提示"小节，给出格式定义 + 阶段映射表；如有循环结构，设计好嵌套格式。
5. **回填改动**：逐处过一遍原文所有确认点，补充"是否受新参数影响"的说明；同步更新 frontmatter description、自检清单、参考文件列表等收尾内容。
6. **通读校对**：检查编号是否连续、文中交叉引用（如"见 Step 0.4""按 Phase 0.1 解析到的值"）在改动后是否依然准确指向正确的位置。

---

## 验收自检清单

- [ ] 有独立的"解析命令行参数"子步骤，放在流程最前面。
- [ ] 所有新增参数都是可选的，不传时行为与改造前一致。
- [ ] 若有"跳过确认"类参数：边界（哪些确认可跳、哪些硬性前提不可跳）已经和用户对齐过，不是自己假设的。
- [ ] 全文所有原有"确认/审批"语句，都补充了是否受新参数影响的说明——不是只在参数定义处写一遍。
- [ ] 有独立的"进度提示"小节，格式统一为 `[第 X/N 步] <阶段名> · <在干什么>`，N 复用文档已有的顶层阶段数。
- [ ] 有一张阶段 → 示例进度行的映射表。
- [ ] 循环结构的进度行能体现"第几个/共几个"。
- [ ] frontmatter description 同步反映了新参数和进度能力。
- [ ] 文档内所有交叉引用编号，在改动后仍准确无误。
- [ ] 除了参数解析和进度提示这两处，没有顺手改动原有的业务判断逻辑。
